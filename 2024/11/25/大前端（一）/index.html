大前端（一）

大前端思想

是相对于传统的前端，iOS，Android，H5等独立小团队而言的，大前端有两个方面的扩展

  • 一个是端的多样性。不局限于Web端，还有IOS、Android、小程序，本来各个端的技术栈不一样，前端无法进行复用，服务端也需要跟每一个前端部门进行对接，整个过程有非常多的重复性工作。大前端采用一些跨平台技术，把各种端融合在一起，这样前端用一套技术栈可以开发出适用于多个平台的客户端，再引入中间层BFF来做适配，对于服务器那边来说，也只需要满足BFF的数据需求就行了，不需要操心各种端的差异。
  • 另外一个就是前端技术的边界扩大化。除了传统的UI开发，还设计到全栈开发、跨平台开发的方向。可能nodejs写后端服务还是没有java这些成熟，但是写BFF这种中间层还是可以胜任的。跨平台开发技术也是因为端的多样性自然而然延伸出来的一些技术。

大前端将不仅仅只是开发web页面，还会做更多的终端开发、工程化工作，成为移动开发与Web前端的发展趋势。

对开发者而言，要会更多的技术栈，比如web开发三剑客，还有Android或iOS的原生开发技术，还有一些跨平台技术等等。

跨平台方案

H5+原生混合开发

微信、淘宝、美团、爱奇艺等app都是用的这种模式。

原理：app需要动态变动的内容通过H5实现，依赖嵌入式浏览器webview这种来加载。H5不能实现的功能，还是交给原生去做，两者之间通过jsbridge桥接技术来进行通信。

js开发+原生渲染

不直接使用 WebView 来渲染界面，而是通过桥接机制将 JavaScript 层的操作转化为原生 UI 组件的操作。

由于 UI 渲染是通过原生组件完成的，性能比 WebView 这种基于 H5 的框架要好，但仍然存在桥接的性能损耗，特别是在复杂的界面和动画中频繁通信会导致卡顿。

React Native

React中虚拟DOM被映射为浏览器DOM树，React Native中虚拟DOM通过JavaScriptCore映射为原生控件树。

JavaScriptCore为js提供了运行环境，也作为js和原生应用通信的桥梁，传递虚拟DOM布局信息。

Weex

思想与RN类似，只是在语法层面，使用的Vue作为开发框架。

快应用

国内9大主流手机厂商共同制定的轻量级应用标准，目标是微信小程序。也是采用js开发+原生渲染。不同的是：

  • 快应用本身不支持vue或react，只支持原生js。
  • 快应用不依赖浏览器引擎了，快应用本身提供了快应用引擎，能直接运行快应用的js、html、css，同时渲染原生控件。不需要嵌入浏览器引擎，也不需要桥接技术，直接用js就能操作原生控件。避免了 WebView 渲染过程中的性能损耗。
  • 快应用的安装包体积较小，因为快应用的渲染引擎是集中到操作系统的ROM中，在应用中无需打包。而React Native 和 Weex 需要在每个应用包中包含渲染引擎和框架，导致应用包体积较大。

自绘UI+原生

通过在不同平台实现一个统一接口的渲染引擎来绘制UI，而不依赖系统原生控件，所以可以做到不同平台UI的一致性。如果涉及其它系统能力调用，依然要涉及原生开发。

优点：可以避免对原生控件依赖而带来的限制及高昂的维护成本。性能高（自绘引擎是直接调用系统API来绘制UI，所以性能和原生控件接近）

缺点：不支持动态下发代码和热更新。

Flutter

Flutter 提供了一个统一的跨平台绘制机制，无论在哪个原生平台上运行，都会通过自己的渲染引擎 Skia 渲染相同的 UI 元素，原生应用只需要作为容器展示 Flutter 渲染的界面，不必担心各平台原生控件的差异，从而确保 UI 的一致性。

PWA

即增强版Web App。PWA完全使用前端技术栈，不过它需要手机和浏览器的支持。

目前支持ServiceWorker和Google Play Service的Android手机，以及搭载11.3以上的iOS手机可以使用PWA。由于国内手机厂商和浏览器厂商的统一性问题，PWA在国内发展不是很好，但是在国外并没有这些问题。
