https

https机制原理

网络传输的安全性

  • http 协议：不安全，未加密
  • https 协议：安全，对请求报文和响应报文做加密

对称加密和非对称加密

对称加密

  • 特点：

    加解密使用 相同 秘钥 高效，适用于大量数据的加密场景 算法公开，安全性取决于秘钥大小，但秘钥越大效率越低，需要权衡在安全和效率中做权衡

  • 缺点：

    算法本身安全，但使用场景不够安全，因为解密和加密都是同一个秘钥。

    算法本身是难解密的，但是在网络传输中因为使用的是同一把钥匙，一开始就需要客户端将密钥传输给服务器，这个过程就有被劫持密钥的风险。

非对称加密

  • 使用 匹配的一对密钥 来分别进行加密和解密，分别是是公开密钥（public key，简称公钥）和私有密钥（private key，简称私钥）
  • 公钥加密的数据 只能 用 对应的私钥解密，同理，私钥加密的数据 只能用 对应的公钥解密
  • 算法：RSA、ECC、Elgamal、背包算法、Rabin、D-H 等
  • 特点：安全性高
  • 缺点：加解密复杂，效率低，耗时较长
  • 用法概要：
    • 加密：对数据做加密（公钥加密，私钥解密）
    • 签名：证明数据是谁发的（私钥加密，公钥能解密就能证明）

加密用法

场景： 用来针对互联网上加密数据传递

  • 准备工作：

    • 生成一对匹配的 私钥 和 公钥 (ps：公钥其实是根据私钥生成的)
    • 将公钥公布给外界

  • 公钥加密：

    • 要发送数据的一方用公钥对数据进行加密，发送

  • 私钥解密

    • 接收到加密数据的一方用自身的私钥进行解密

    因为一个公钥加密的数据只有对应的私钥才能解密，所以密文很安全

    补充：如果要在网络上相互发送密文，可以让对方也发对方的公钥过来，用对方的公钥来加密

私钥签名

场景： 目的是为了将明文公布给别人，同时证明是自己发的；可以防止明文被篡改。

以A向B发送”哈哈哈“为例解析私钥签名的过程：

  • A用自己的私钥对“哈哈哈”明文的hsah值进行加密形成签名，然后把签名和明文一并发送给B。

由于明文一般较长，故进行非对称加密的效率较差。hash以后得到的是固定长度的信息，可以提升加密效率。

  • B用A的公钥对接收到的签名进行解密，对接收到的明文进行hash，通过对比解密后得到的明文hash值和接收到的明文hash值判断

https协议安全传输原理

HTTPS是运行在SSL/TLS之上的 HTTP 协议，所有传输的内容都经过加密，加密采用的是对称加密，但对称加密的密钥用服务器证书进行了非对称加密。

数字证书

非对称加密之所以不安全，是因为客户端收到公钥之后，无法确定这把公钥是否是来自服务端。无法确定这把公钥是否是别人冒充的。

因此引入了拥有公信力的认证中心CA生成数字证书。服务器会用CA提供的私钥对消息摘要进行加密来形成数字签名。

并且，最后还会把原来没Hash算法之前的信息和数字签名合并在一起，形成数字证书。

当客户端拿到这份数字证书之后，就会用CA提供的公钥来对数字证书里面的数字签名进行解密得到消息摘要，然后对数字证书里面小服的公钥和个人信息进行Hash得到另一份消息摘要，然后把两份消息摘要进行对比，如果一样，则证明这些东西确实是小服的，否则就不是。

小结

https保障内容的安全传输的本质还是使用对称加密算法，但是对称加密的秘钥容易被窃取，因而又使用了非对称加密算法对秘钥进行加密，又为了保障非对称加密的公钥不被冒充（伪造），引入了数字证书。

  • CA认证中心会生成数字证书，数字证书包含两个部分，一个是服务器的公钥和基本信息，另一个是经过认证中心对这些内容的消息摘要进行私钥加密后形成的数字签名。
  • 当客户端拿到这份数字证书之后，会查看操作系统的证书拿到这个数字证书的CA公钥，来对数字证书里面的数字签名进行解密验证，如果验证通过就可以拿到这个确保数字证书里存放的服务器公钥不是伪造的。
  • 确保拿到了服务器的公钥之后，浏览器就可以放心的把会话秘钥通过公钥以非对称加密的方式传递给服务器。
  • 服务器收到后，用私钥解密，获取到了浏览器的会话秘钥。
  • 后续只需要使用共同的会话秘钥通过对称加密的方式，进行传输内容了
  • 既保证了传输的安全性，又避免了一直使用非对称加密传输数据太大的问题
