Webpack原理

Webpack究竟解决了什么问题？

前端应用的繁荣发展，项目代码不断膨胀.。为了减少维护成本，提高代码的可复用性，在前端项目中更高效的管理和维护项目中的每一个资源 ==> 模块化思想=>模块化演变=>模块化规范=>模块化打包工具

模块化演变

早在模块化标准还没有诞生的时候，前端模块化的雏形有三种形式：

  • 文件划分

  • 命名空间

  • IIFE 私有作用域

文件划分：

把应用的状态和逻辑放到不同的 JS 文件中，HTML 中通过不同的 script 标签一一引入。这样的缺点是：

  1. 模块变量相当于在全局声明和定义，会有变量名冲突的问题。
  2. 变量都在全局定义，导致难以调试，我们很难知道某个变量到底属于哪些模块。
  3. 无法清晰地管理模块之间的依赖关系和加载顺序。假如 a.js 依赖 b.js，那么 HTML 中的 script 执行顺序需要手动调整，不然可能会产生运行时错误。

命名空间和IIFE（立即执行函数）都可以解决文件划分的前两个缺点，但同样的存在第三个缺点，以及代码不优雅等问题。

之后开始出现一些模块化的规范。

模块化规范

CommonJS

是Node.js中遵循的模块规范，约定

  • 一个文件就是一个模块
  • 每个模块都有单独的作用域
  • 通过module.exports 导出成员
  • 通过require函数载入模块

CommonJS 定义了一套完整的模块化代码规范，不过仍然存在一些问题：

  • 它的模块加载器由 Node.js 提供，依赖了 Node.js 本身的功能实现。如果 CommonJS 模块直接放到浏览器中无法执行。
  • CommonJS 约定以同步的方式进行模块加载，这种加载机制放到浏览器端，会带来明显的性能问题。它会产生大量同步的模块请求，浏览器要等待响应返回后才能继续解析模块。即，模块请求会造成浏览器 JS 解析过程的阻塞，导致页面加载速度缓慢。

因此，该规范主要适用于服务端，因为服务端可以直接从硬盘中调用所需要的模块，而这个过程速度是非常快的。

典型代表有：Node.js、微信小程序。

但是使用webpack以后中可以在浏览器中使用commonjs，因为webpack可以对其进行处理。

ES Module

在浏览器中使用ES Module模块化规范

导出

导入

  • export导出不是把值复制一份给你，而是把存放值的地址给你（拿到的值会受内部值修改的影响），外部只读不可修改。（浅复制）
  • 导入的成员必须要有完整的文件名称，相对路径中的./不能省略，可以使用绝对路径或完整的url（ /xx/modules.js 或 http:// ）

使用相对路径的话，不能省略./

若只需要执行模块，而不提取成员

要拿的成员很多可以先放到一个对象里

如果需要动态加载模块

运行环境兼容性问题

由于ES Module是2014年提出来的，早期的浏览器不支持这一特性。兼容性问题还是需要考虑

模块化打包工具的出现

问题：

  • ES Modules模块系统本身存在环境兼容问题
  • 模块化的方式划分出来的模块文件过多，导致浏览器需要频繁发送网络请求，影响应用工作效率
  • HTML和CSS这些资源文件也会面临需要被模块化的问题

需要一些解决方案/工具可以让我们在开发应用的过程中继续享受模块化带来的优势，又不必担心模块化对生产环境的影响。

需求：

  • 具备编译能力，可以将开发阶段编写的包含新特性的代码转化为能够兼容大多数环境的代码 —— 解决环境兼容问题

  • 将散落的模块打包到一起，解决浏览器频繁请求模块文件、发送网络请求的问题 —— 解决请求频繁问题

  • 支持不同种类的前端资源模块类型（css,png,…） —— 解决不同资源文件模块化问题

    前两点gulp也可以实现，但第三点gulp很难实现，所以前端主流打包工具——Webpack

如何使用Webpack进行模块化打包？

本质：实现前端整体的模块化开发的一个模块化打包工具。

作为一个模块打包器，webpack具有以下特性

  • 模块加载器（Loader） 编译转换有环境兼容问题的代码 ( 比如转换源代码，解决有些浏览器不兼容ES Module)
  • 代码拆分（Code Splitting） 解决打包文件太大、太碎的问题。比如可以把一开始用到的模块打包到一起，其他模块等到需要时再加载，实现渐进式加载
  • 资源模块（Asset Module） webpack支持在js以模块化的方式载入任意类型的资源文件,比如css类型文件

快速上手

1.不使用webpack打包工具

ES Module的一种载入方式，type="module"用来区分加载的是一个普通JS脚本还是一个模块

对于不支持ES Module的浏览器，直接这样使用就会报错

2.使用webpack打包工具

  • 初始化
    • webpack本身是npm的一个工具模块，首先初始化package.json用来管理npm依赖的版本
    • webpack是Webpack的核心模块，webpack-cli是Webpack的CLI程序，用来在命令行中调用Webpack

npx是npm5.2以后新增的一个命令，可以更方便的执行远程模块或者node_modules中的CLI程序

  • 运行

Webpack默认会自动从src/index.js文件开始打包，然后根据代码中的模块导入操作自动将所有用到的模块打包到一起。

  • dist

打包结果存放在项目根目录下的dist目录下的main.js文件中

==3.修改webpack的默认配置==

entry（入口）: 指示 Webpack 从哪个文件开始打包

output（输出）: 指示 Webpack 打包完的文件输出到哪里去，如何命名等

loader（加载器）: webpack 本身只能处理 js、json 等资源，其他资源需要借助 loader，Webpack 才能解析

plugins（插件）: 扩展 Webpack 的功能

mode（模式）: 主要有两种模式，development（开发模式）和 production（生产模式）

格式：

  • 在项目根目录下添加一个webpack.config.js配置文件

    这个文件运行在nodejs环境中，需要按照CommonJS的规范导入模块导出成员，可以直接在这个文件中使用path之类的Node.js内置模块

如何通过Loader实现特殊资源加载？

Loader 专注实现 资源模块的加载，从而实现整体项目的打包。

webpack内部默认的loader只能处理js模块，如果需要加载其他类型的模块，就需要配置不同的loader

以加载css资源模块为例

  • 通过npm先去安装这个Loader

  • 在配置文件添加对应的配置——css-loader

  • 安装style-loader

    由于css-loader只会把css模块加载到JS代码中，但并不会使用这个模块,需要style-loader将css-loader转换后的结果通过style标签的方式追加在页面。

如何利用插件机制横向扩展Webpack的构建能力？

Plugin 解决项目中 除了资源加载以外 的自动化工作。

应用场景

  • 实现自动在打包之前清除dist目录（上次的打包结果）
  • 自动生成应用所需要的HTML文件

即自动生成一个使用所有打包结果的html文件到dist目录下。自动引入资源文件的方式，可以确保文件路径正确。

  • 根据不同环境为代码注入类似API地址这种可能变化的部分
  • 拷贝不需要参与打包的资源文件到输出目录

有一些不需要参与构建的静态文件，最终也需要发布到线上，建议把这类文件统一 放在项目根目录下的public或者static目录中。

  • 压缩Webpack打包完成后输出的文件
  • 自动发布打包结果到服务器实现自动部署

插件机制实现原理

处理 js 兼容性和规范性

有人可能会问，js 资源 Webpack 不能已经处理了吗，为什么我们还要处理呢？

原因是 Webpack 对 js 处理是有限的，只能编译 js 中 ES 模块化语法，不能编译其他具有新特性的js语法。

其次开发中，团队对代码格式是有严格要求的，我们不能由肉眼去检测代码格式，需要使用专业的工具来检测。

  • 针对 js 兼容性处理，我们使用Babel来完成
  • 针对代码格式，我们使用 Eslint 来完成

我们先完成 Eslint，检测代码格式无误后，在由Babel做代码兼容性处理

Eslint

用来检测 js 和 jsx 语法的工具，可以配置各项功能。(Plugin)

1.下载包

2.定义Eslint配置文件.eslintrc.js（位于项目根目录）

3.将Eslint配置到webpack中

4.运行指令，在控制台查看Eslint检查效果

5.可以通过配置.eslintignore更精细的控制Eslint检查、不检查的文件

Babel

Babel是目前主流的一款js编译器，可以帮助我们将一些使用了新特性的代码编译为当前环境支持的代码。

Babel是基于插件机制实现的，我们需要一个插件来转换一个代码特性，preset-env相当于一个插件的集合，这个集合包含了最新的js新特性，

1.下载包

2.定义Babel配置文件babel.config.js

  • presets 预设：就是一组 Babel 插件, 扩展 Babel 功能
    • @babel/preset-env: 一个智能预设，允许您使用最新的 JavaScript。
    • @babel/preset-react：一个用来编译 React jsx 语法的预设
    • @babel/preset-typescript：一个用来编译 TypeScript 语法的预设

3.将Babel 配置到webpack中

Webpack核心工作原理

Webpack在整个打包的过程中：

  • 通过Loader处理特殊类型资源的加载，例如加载样式、图片
  • 通过Plugin各种自动化的构建任务，例如自动压缩、自动发布

查阅源码的思路：

  1. Webpack CLI启动打包流程

    Webpack CLI：作用就是将CLI参数和Webpack配置文件中的配置整合得到一个完整的配置对象

    Webpack CLI会通过yargs模块解析CLI参数(运行webpack命令时通过命令行传入的参数)，调用bin/utils/convert-argv.js模块将得到的命令行参数转换为Webpack的配置选项对象。然后开始载入Webpack核心模块，传入配置选项，创建Compiler对象。

  2. 载入Webpack核心模块，创建Compiler对象

    根据传入的Option是数组还是对象，决定多路打包还是单线打包。

  3. 使用Compiler对象开始编译整个项目

    创建了Compiler之后Webpack开始注册已配置的插件，触发特定的Hook,调用Compiler对象的run方法，开始构建整个应用。

  4. 从入口文件开始，解析模块依赖,形成依赖关系树

    创建compilation作为整个上下文变量，紧接着触发make钩子，进入到整个构建过程中最核心的阶段：make阶段。

  5. 递归依赖树，将每个模块交给对应的Loader处理

    make阶段主体的目标：根据entry配置找到入口模块，开始依次递归出所有依赖，形成依赖关系树，然后将递归到的每个模块交给不同的Loader处理 。

  6. 合并Loader处理完的结果，将打包结果输出到dist目录
